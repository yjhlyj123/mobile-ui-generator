# 设计引导流程

> 本文件定义 Agent 如何引导没有 UI 设计师的用户，通过结构化交互产出令人满意的界面设计。
> 核心原则：**先了解 → 再推荐 → 迭代确认 → 一致性保障**。

---

## 阶段 1：需求收集（自动问卷）

当用户首次进入复杂模式或设计先行模式时，Agent 通过以下结构化问答收集信息。对于用户未提供的项目，使用默认假设并明确标记。

### 必问项

| 问题 | 示例答案 | 默认假设 |
|------|----------|----------|
| **行业/业务类型** | 电商、OA、教育、医疗、金融 | B 端企业应用 |
| **目标人群** | C 端消费者 / B 端企业用户 / 内部员工 | B 端企业用户 |
| **核心页面列表** | 首页、工作台、消息、我的 | — |
| **是否有 Tabbar** | 是 / 否 | 否 |

### 选问项（视情况询问）

| 问题 | 选项 | 默认 |
|------|------|------|
| **风格偏好** | 简约 / 活力 / 商务 / 科技 / 温暖 | 商务 |
| **品牌色** | 用户提供 hex 值 | 商务蓝 `#0E74FF` |
| **参考 App** | 钉钉、飞书、微信、支付宝… | 无 |
| **参考截图** | 用户上传 | 无 |
| **特殊要求** | 暗色模式、无障碍、国际化… | 无 |

### 收集规则

1. **不要一次问太多**：首轮最多问 3-4 个核心问题
2. **有默认假设**：用户不回答的项按默认假设处理，并在方案中标记
3. **渐进式深入**：先了解大方向，再在方案确认后追问细节

## 阶段 2：风格推荐（AI 策略）

### 匹配矩阵

根据收集的行业 + 人群 + 偏好，从预设主题和 Tabbar 风格中匹配推荐：

| 行业 + 人群 | 推荐主题 | 推荐 Tabbar 风格 | 推荐头部处理 |
|------------|----------|------------------|-------------|
| B 端 + OA/审批 | 商务蓝 | 沉浸式头部 / 卡片金刚区 | 渐变背景 |
| B 端 + 数据管理 | 商务蓝 | 数据看板式 | 简洁白底 |
| C 端 + 电商 | 活力橙 | 图文并茂式 | 图片背景 |
| C 端 + 教育 | 清新绿 | 图文并茂式 | 品牌插图 |
| C 端 + 健康 | 清新绿 | 沉浸式头部 | 渐变背景 |
| B 端 + 金融 | 商务蓝 | 数据看板式 | 渐变背景 |

### 方案生成策略

在生成方案前，Agent 必须基于阶段 1 收集的信息，先内部推导以下三点（不输出给用户）：

1. **用户的核心任务是什么**：这个页面的用户打开它最想完成什么？（如：快速处理待办、查看今日数据、找到某个功能入口）
2. **信息的优先级排序**：哪些信息必须首屏可见，哪些可以折叠或下沉？
3. **高频操作 vs 低频操作**：用户每天会反复做的事 vs 偶尔才用的功能

然后围绕这三点，设计 3 个在**信息架构层面有本质差异**的方案。三个方案应分别代表不同的设计假设，例如：
- 方案 A 假设用户最关心"待处理事项"，首屏以待办列表为主角
- 方案 B 假设用户最关心"数据全局"，首屏以统计概览为主角
- 方案 C 假设用户最关心"快速操作"，首屏以功能入口为主角

**禁止**仅通过换颜色、换头部样式、换装饰元素来制造方案差异。

### 方案输出

使用 `core/ui-direction-template.md` 的格式，为用户输出至少 3 个方案。每个方案需包含：

1. **一句话定位**：说明该方案假设用户最想做什么
2. **用户动线**：从打开页面到完成核心任务的路径
3. **页面布局示意**：ASCII 图（标注具体业务内容，不要只写"模块1"）
4. **首屏信息分配**：每个区域放什么、为什么放在这里
5. **选择建议**：什么情况下适合选这个方案、什么情况下不适合

方案对比总结中必须包含**具体的选择指引**（"如果你的用户主要是 X，推荐选方案 A"），不得只说"请选择"。

## 阶段 3：方案选择与迭代

### 交互循环

```
Agent 输出 3 个方案
     ↓
用户反馈（选择 / 不满意 / 微调）
     ↓
  ┌─ 选择方案 → 进入阶段 4（Pencil 视觉设计）
  ├─ 微调 → Agent 调整后重新呈现
  └─ 不满意 → Agent 提供 3 个新方案（可询问更具体的偏好）
```

### 关键规则

1. **用户未选定前不得写代码**：严格遵守命令契约的强制停顿点
2. **选定方案后不得直接写代码**：必须先进入阶段 4 的 Pencil 视觉设计流程
3. **支持局部微调**："保留方案 B 的头部，但用方案 A 的金刚区"
4. **记录选择**：确认后将用户选择的方案信息记录到结构化 Prompt 中
5. **最多 5 轮**：如果 5 轮后仍未达成一致，建议用户提供更明确的参考

## 阶段 4：Pencil 视觉设计与确认

> **核心原则：用户必须先看到视觉效果，反复修改满意后，才能进入代码实现。**
> **选定方案 ≠ 可以写代码。选定方案只是确定了设计方向，用户还没看到实际的视觉效果。**
> **本阶段适用于所有非 simple 流程，包括 complex、design_first 和 refactor 模式。**

### 完整流程

```
用户选定方案（阶段 3）
     ↓
Agent 编写结构化 Prompt
  ├─ 普通模式：基于选定方案 + prompt-rules.md + pencil-prompt-template.md 标准模板
  └─ 重构模式：基于选定重构方向 + pencil-prompt-template.md 的 Refactor Mode Template
     ↓
Agent 调用 Pencil 生成视觉设计稿
     ↓
Agent 展示设计稿给用户
     ↓
用户反馈（满意 / 需要调整）
     ↓
  ┌─ 满意 → 用户明确说"可以写代码了" → 进入阶段 5（代码实现）
  ├─ 局部调整 → Agent 修改 Prompt 并重新调用 Pencil → 再次展示
  └─ 大改 → 回到阶段 3 重新选方案
```

### 强制规则

1. **必须调用 Pencil**：选定方案/方向后，Agent 必须将其转化为结构化 Prompt 并调用 Pencil 生成视觉设计，不得跳过
2. **必须展示设计稿**：Pencil 生成的设计稿必须展示给用户查看
3. **必须等待明确同意**：用户必须明确表示对设计稿满意（如"可以了"、"开始写代码"、"确认"），Agent 才能进入代码实现
4. **不设迭代上限**：用户可以要求修改设计稿任意次数，Agent 每次修改后都要重新调用 Pencil 并展示。Agent 不得主动建议用户跳过设计环节
5. **绝对禁止**：
   - 用户选完方案后直接写代码
   - 跳过 Pencil 设计直接实现
   - 没有得到用户对设计稿的明确同意就开始写代码
   - 把"用户选了方案"等同于"用户同意了设计"

### 重构模式特别说明

重构模式同样必须走 Pencil 设计流程。用户提供了现有页面代码，意味着对原有 UI 不满意，更需要先看到新设计的视觉效果再决定是否转为代码。

- 重构方向选定后，使用 `pencil-prompt-template.md` 的 **Refactor Mode Template** 编写结构化 Prompt
- Prompt 中必须明确列出"从参考页提取的业务内容"和"禁止复现的参考页元素"
- Pencil 设计稿需体现与原有页面的差异

### Pencil 不可用时的处理

如果 Pencil 连接失败或不可用：
- 不得自动降级为直接写代码
- 必须先尝试安装/修复 Pencil（参见 workflow.md 的 Pencil 安装流程）
- 修复失败时，停在此阶段，向用户说明阻塞原因
- 只有用户明确说"跳过设计，直接写代码"时，才可以绕过此阶段

## 阶段 5：一致性保障

### 选定方案后的约束

一旦用户选定方案，后续所有页面开发必须遵守：

1. **主题锁定**：使用选定的主题预设（如 orange.json），所有 Token 从该预设取值
2. **风格锁定**：一级页面使用选定的 Tabbar 风格，二级页面使用标准布局
3. **自动检测**：新页面开发时，Agent 应主动对照 `design-spec.md` 自检清单
4. **差异提醒**：如果新页面有偏离已选方案的风险，Agent 应提前提醒

### 样式统一要素

| 要素 | 一致性要求 |
|------|-----------|
| 品牌色 | 所有页面使用同一 `--mug-color-primary` |
| 圆角规格 | 一级页面 `$radius-xxl`，二级页面 `$radius-md` |
| 阴影层级 | 按 `design-spec.md` 规范使用 |
| 字体层级 | 严格遵循 Token 定义的字号 / 字重 |
| 空状态 | 使用统一的空状态插图风格 |
| 加载态 | 使用统一的骨架屏或 Loading 组件 |
