# 移动端 UI 生成器 — 核心 Prompt 规则 v4.0

> **注意：所有视觉设计规范、Tokens 和组件用法请参阅：`design-spec.md`。**

## 1. 结构化 Prompt 核心要素 (针对所有非 simple 流程)

在调用 Pencil 或生成复杂代码前，必须先将用户需求整理为以下结构：

- **方向选择前置**：先提出至少 3 个差异化 UI 方案，让用户选择；未选定前不得直接进入实现。
- **Pencil 设计必经**：用户选定方案后，必须编写结构化 Prompt 并调用 Pencil 生成设计稿，展示给用户确认；用户未对设计稿明确满意前，不得进入代码实现。选定方案 ≠ 同意设计。
- **业务场景**：页面所属的业务模块（如：电子签章、审批流、OA首页）。
- **核心模块**：按优先级列出页面必须包含的功能块（如：状态看板、待办列表、操作金刚区）。
- **内容约束**：坚持信息克制原则，不添加弱价值文案。
- **技术栈**：`uni-app + Vue 3 + UnoCSS + wot-design-uni`。

## 2. Pencil 专属规则

使用 Pencil 设计时，应重点描述：
- **容器结构**：使用 `Section`、`Card` 等语义化布局描述。
- **信息密度**：明确每个模块的信息展示深度。
- **交互锚点**：定义按钮、搜索框、筛选器的位置。
- **微信胶囊适配**：一级页面必须标注胶囊区域安全距离。

## 3. 设计规范引用

禁止在 Prompt 中重复定义颜色、圆角、间距等具体数值。只需引用规范描述：
- "遵循 `design-spec.md` 中的标准卡片布局"
- "使用 `design-spec.md` 定义的业务状态色映射"
- "采用 `design-spec.md` 的筛选器规范"
- "表单布局遵循 `design-spec.md` § 5"
- "列表操作按钮遵循 `design-spec.md` § 6.4"
- "详情页分组遵循 `design-spec.md` § 7"

## 4. 重构模式 Prompt 策略

当用户要求"重构"或提供参考图/现有代码时：
- **提取**：仅提取参考页的业务模块、数据字段、状态和操作。
- **重组**：彻底打破原有布局，按 `design-spec.md` 的规范重新排布。
- **差异化**：至少提供 3 个不同侧重的布局方案（如：极致精简 vs 模块分区 vs 状态前置）。
- **Pencil 必经**：重构模式同样必须走 Pencil 设计流程，用户对原 UI 不满意，更需要先看到新设计效果。

## 5. 表单页面 Prompt 规则

当页面类型为表单（`/mug:*:form`）时，结构化 Prompt 必须包含：

- **字段清单**：列出所有字段及其类型（文本 / 数字 / 选择 / 日期 / 开关 / 上传）
- **必填标注**：每个字段标注必填 / 选填
- **校验规则**：每个必填字段的校验条件（如：不为空、格式校验、长度限制）
- **联动逻辑**：字段间的显示/隐藏/值联动关系（如："合同类型"选择"采购"时显示"供应商"字段）
- **字段分组**：相关字段的分组方式和组名
- **提交操作**：按钮数量（1 个提交 / 2 个草稿+提交）和位置（底部固定）
- **Pencil 模板**：使用 `pencil-prompt-template.md` 的 Form Page Template

### 5.1 表单校验自动推断规则

当生成表单代码时，Agent 必须：

1. **自动推断校验类型**：根据字段名关键词匹配 `design-spec.md` §5.7 映射表，自动添加对应的 `wd-form` rules 配置
2. **自动匹配键盘类型**：根据推断的校验类型，设置对应的 `wd-input` type 属性（如手机号 → `type="number"`，身份证 → `type="idcard"`，金额 → `type="digit"`）
3. **非必填也校验格式**：即使字段非必填，只要匹配到格式规则，用户有输入时也要校验格式正确性
4. **必填提示区分**：输入类用"请填写{label}"，选择类用"请选择{label}"，上传类用"请上传{label}"
5. **不需要用户逐一指定**：用户只需提供字段名和类型，校验规则由 AI 根据 §5.7 映射表自动推断补全
6. **联动字段**：如果用户描述了字段间的依赖关系，必须在代码中实现联动逻辑，遵循 `design-spec.md` §5.8 联动规范
7. **长表单处理**：字段数 > 6 时自动分组，> 12 时考虑折叠，> 20 时建议分步，遵循 `design-spec.md` §5.9

## 6. 列表页面 Prompt 规则

当页面类型为列表（`/mug:*:list`）时，结构化 Prompt 必须包含：

- **列表项结构**：明确主标题 / 副信息 / 状态 / 时间 / 操作按钮的字段映射
- **操作按钮**：列出每个按钮的名称和类型（普通 / 危险），标注按钮数量
- **操作模式**：根据按钮数量和频率选择模式 A（内联+更多）或模式 B（左滑），并说明选择原因
- **搜索维度**：可搜索的字段（如：名称、编号）
- **筛选维度**：筛选条件及选项（如：状态 [全部/待审批/已通过]）
- **状态处理**：空数据、搜索无结果、加载失败的处理方式
- **Pencil 模板**：使用 `pencil-prompt-template.md` 的 List Page Template

### 6.1 列表数据展示自动推断规则

当生成列表代码时，Agent 必须：

1. **自动推断展示方式**：根据字段名关键词匹配 `design-spec.md` §6.7 映射表，自动决定每个字段的展示方式（StatusTag / 金额格式化 / 脱敏 / 进度条等）
2. **自动选择布局变体**：根据字段组成匹配 `design-spec.md` §6.8 变体选择规则（标准 / 密集 / 缩略图 / 进度 / 简洁）
3. **自动格式化数据**：所有数据字段必须使用 `design-spec.md` §6.6 格式化规则，禁止模板内联格式化
4. **空值统一处理**：所有字段必须兜底空值，统一显示 `--`
5. **主标题自动识别**：含 name/title/subject 的字段自动作为列表项主标题
6. **副信息自动组合**：人名、部门、时间等字段自动用 `·` 分隔组合为副信息行

## 7. 详情页面 Prompt 规则

当页面类型为详情（`/mug:*:detail`）时，结构化 Prompt 必须包含：

- **信息分组**：按业务语义分组，标注每组的字段列表和展示优先级
- **状态字段**：标注状态字段名称及对应的状态色（使用四层状态色体系）
- **关键字段**：需要视觉强调的字段（如金额使用加粗加大）
- **时间线**：如有审批记录/操作日志，标注使用时间线展示
- **底部操作**：按钮数量、每个按钮的名称和类型（主操作 / 次操作 / 危险操作）
- **Pencil 模板**：使用 `pencil-prompt-template.md` 的 Detail Page Template

### 7.1 详情数据展示自动推断规则

当生成详情页代码时，Agent 必须：

1. **自动推断展示方式**：根据字段名关键词匹配 `design-spec.md` §7.4 映射表，自动决定每个字段的展示方式和交互行为
2. **自动构建头部层级**：根据 `design-spec.md` §7.5 规范，自动构建三层信息层级（标题+状态 → 关键指标 → 元信息）
3. **可交互字段自动添加操作**：phone → 拨号图标、email → 邮件链接、url → 跳转、编号 → 复制图标
4. **附件字段自动使用附件规范**：推断为文件/图片的字段，自动使用 `design-spec.md` §7.6 附件展示规范
5. **关联数据自动限制**：嵌套列表最多展示 3 条 + "查看全部"链接，遵循 `design-spec.md` §7.7
6. **格式化必须调用**：所有数据字段必须使用 `design-spec.md` §6.6 格式化规则

## 8. 通用交互 Prompt 规则

### 8.1 页面导航规则

Agent 生成代码时必须：

1. **列表→详情**：使用 `uni.navigateTo`，query 只传 `id`，详情页自行请求完整数据
2. **表单提交后返回**：`uni.$emit('list-refresh')` + `uni.navigateBack()`，不能只返回不通知
3. **列表页刷新**：`onLoad` 中 `uni.$on('list-refresh', ...)`，`onUnload` 中 `uni.$off` 清理
4. **编辑模式标识**：表单页通过 `mode=create|edit` 区分，编辑模式需请求数据回填
5. **参见规范**：`design-spec.md` §8.12

### 8.2 错误处理规则

Agent 生成代码时必须：

1. **表单提交 try/catch**：失败时 toast 具体错误，不清空用户输入，`finally` 中恢复按钮
2. **列表请求 try/catch**：首次失败显示 `EmptyState` type=load-failed，加载更多失败显示 `wd-loadmore` error 态
3. **按钮 loading 防重复**：操作前 `loading=true`，`finally` 中恢复
4. **禁止空 catch**：catch 中必须给用户反馈（toast 或 EmptyState）
5. **参见规范**：`design-spec.md` §8.13

### 8.3 反馈文案规则

Agent 生成代码时必须：

1. **Toast 文案格式**：成功用 `{动词}成功`，失败用 `{动词}失败，{原因}`
2. **确认弹窗文案格式**：`确定{动词}{对象}？`（如"确定删除该合同？"）
3. **禁止笼统文案**：不允许"操作成功"、"操作失败"、"系统错误"、"确定？"
4. **Toast 时长**：success `1500ms`，error/warning `2000ms`
5. **参见规范**：`design-spec.md` §8.2

### 8.4 骨架屏规则

Agent 生成代码时必须：

1. **匹配页面类型**：表单页用表单骨架、列表页用列表骨架、详情页用详情骨架
2. **禁止通用骨架**：不允许所有页面使用同一个 `wd-skeleton` 配置
3. **参见规范**：`design-spec.md` §8.3.1

## 9. 反 AI 生成感规则

所有 Pencil 设计和代码实现必须避免"AI 批量生成"的模板感。以下为强制执行的具体规则：

### 8.1 布局节奏

- **禁止全页等分网格**：不做 2×2、3×3 的完美对称卡片网格，区块应有大小主次之分
- **要求节奏变化**：页面从上到下，区块大小应交替变化（如：大卡片 → 小列表 → 中等信息组），不是均匀排列
- **首屏必须有视觉重心**：一个明确的"主角区块"占据首屏主要面积，辅以 1-2 个次要区块，不是均匀分布

### 8.2 排版层次

- **标题与正文字号差 ≥ 4px**：要有明显的大小对比，不能所有文字视觉大小接近
- **同一卡片内最多 3 级字号**：标题 / 正文 / 辅助信息，不要出现 4-5 种字号
- **粗细对比**：标题用 `$font-weight-medium` 或 `$font-weight-bold`，正文用 `$font-weight-regular`，形成明确的粗细层次

### 8.3 内容真实感

- **使用具体业务数据**：设计稿和代码中使用"¥125,000.00"、"张三"、"采购合同-2024001"等真实感数据，不用"标题文本"、"描述信息"、"用户名"
- **状态要有语义**：使用"待审批"、"已生效"、"已驳回"等具体状态词，不用"状态1"、"标签"
- **数量要合理**：列表展示 3-5 条不同的真实数据，不是完全相同的重复内容

### 8.4 色彩使用

- **大面积灰白 + 局部点睛**：页面 80%+ 面积应为灰白（`$color-bg-page`、`$color-bg-card`），品牌色仅用于按钮、状态标签、关键指标等点睛位置
- **不要处处有色彩**：禁止每个卡片都有彩色背景或彩色图标底色
- **渐变仅限一级页面头部**：二级页面不使用渐变

### 8.5 禁止的 AI 套路

以下为典型的 AI 生成设计套路，必须明确避免：

| 套路 | 具体表现 | 替代方案 |
|------|----------|----------|
| **万能模板首页** | 渐变头部 + 问候语 + 2×2 金刚区 + 列表 | 根据业务场景设计差异化首屏，金刚区不一定需要 |
| **完美对称网格** | 所有卡片等大等高、等间距排列 | 区块有大小之分，关键区块更大 |
| **彩虹图标** | 每个图标配一个不同颜色的圆形/方形底色 | 图标底色最多 2-3 种，或统一使用品牌色 |
| **万能数据面板** | 2×2 数据卡片 + 上下箭头趋势 + 百分比 | 根据实际数据类型选择展示方式，不必每个都有趋势 |
| **通用问候区** | "欢迎回来，XXX" + 圆形头像 + 日期 | 如确需问候，结合业务信息（如"你有 3 条待审批"） |
| **装饰性半透明圆** | 头部背景散落半透明圆形/方形装饰 | 装饰元素应有功能含义，或直接不加 |
| **千篇一律空状态** | 居中灰色图标 + "暂无数据" | 空状态应说明原因和下一步操作 |
