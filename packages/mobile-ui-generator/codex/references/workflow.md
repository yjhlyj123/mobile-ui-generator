# Codex 适配器：工作流

> 命令解析、组合优先级、缺失输入矩阵与状态机的单一事实源：`../../core/command-contract.yaml`

在 Codex 中使用本适配器流程。

## 1. 判断页面复杂度

### 按页面类型判断

- 简单页面：标准表单、普通列表、普通详情页、基础搜索页、空状态页、普通结果页
- 复杂页面：工作台首页、数据看板、审批流编排页、多分区复合首页、带自定义导航的业务首页、引用了已有 Pencil 草稿或 `.pen` 风格的页面

### 按需求内容判断（用户提供功能描述或接口信息时使用）

**命中任意 2 条及以上 → 复杂页面：**

| 信号 | 判断依据 |
|---|---|
| 接口数量 ≥ 3 个 | 需要同时调用多个接口聚合数据 |
| 存在多个独立业务模块 | 如「待办 + 快捷入口 + 动态列表」并存于同一页面 |
| 存在状态流转或审批流 | 状态之间有跳转逻辑 |
| 存在自定义导航或 tabbar | 非标准页面结构 |
| 首屏需展示 3 类以上不同性质的数据 | 如统计数字 + 列表 + 操作卡片同屏显示 |
| 用户角色或权限影响页面内容 | 不同角色看到的模块或操作不同 |
| 页面包含图表、进度环、统计卡片 | 数据可视化组件 |
| 需求描述超过 5 个独立功能点 | 功能点数量反映页面信息承载量 |

**全不命中 → 简单页面，直接生成。**

## 2. 选择路径

- 简单页面：直接按 skill 规则生成
- 复杂页面或设计先行页面：若 Pencil MCP 可用，必须先给出至少 3 个差异化 UI 方向，再让用户选择
- 用户选定方向后：再整理对应方向的 Pencil prompt，并进入实现
- Pencil 不可用：停止当前开发，只输出诊断结果、失败原因和下一步需要用户执行的动作
- 具体状态流顺序以 `../../core/command-contract.yaml` 为准

### ⚠️ 计划模式特别说明

当 Codex 进入计划模式（plan mode）时，计划中 **必须** 包含以下步骤，不得省略或合并：

1. **Pencil 安装/验证步骤**：计划中必须包含检测和验证 Pencil MCP 连接的步骤
2. **UI 方案输出步骤**：计划中必须包含"输出至少 3 个差异化 UI 方案并等待用户选择"的步骤
3. **Pencil 设计生成步骤**：计划中必须包含"调用 Pencil 生成设计稿"的步骤
4. **用户确认设计步骤**：计划中必须包含"展示设计稿并等待用户明确确认"的步骤
5. **代码实现步骤**：只有在计划中明确标注"用户确认设计后"才能安排代码实现步骤

**绝对禁止**：在计划中跳过 Pencil 设计步骤直接安排代码实现。计划模式不是绕过设计流的理由。

### 2.5 项目上下文（多页面一致性）

Agent 通过项目根目录的 `.mug-project.json` 文件在多个页面间保持风格一致。

- **首个页面**：完成设计选择后，Agent 必须将决策（主题、品牌色、风格方向、Tabbar 风格）写入 `.mug-project.json`
- **后续页面**：开发新页面时，先读取 `.mug-project.json`，沿用已有风格决策，仅询问页面特定细节
- **用户覆盖**：用户可随时要求重新选择风格，Agent 更新 `.mug-project.json`

## 3. 重构模式

当用户提供参考页面（截图、`.pen` 文件或页面描述）并要求重构另一个页面时，进入重构模式。

- 参考页面只作为**内容来源**：提取模块、数据字段、业务状态和操作入口
- **不得**照搬参考页面的布局、卡片形态、色块用法或信息分组
- 在实现任何代码之前，始终给出 **3 个差异化方向**

**方向一：精简聚焦** — 折叠次要信息，首屏只保留核心操作与关键状态，单列平铺

**方向二：模块分区** — 内容划分为独立卡片或分区块，每块有明确标题，均衡密度

**方向三：状态前置** — 状态标签、指标、进度前置首屏，信息密度更高，减少跳转

给出 3 个方向后，请用户选择再实现。如果用户已指定方向，直接实现。
最终页面必须在视觉和结构上与参考页面明显不同。

## 3.5 `.pen` 文件处理

进入 Pencil 设计步骤时：
- 若用户提供了 `.pen` 文件，使用 `open_document` 打开该文件
- 若用户未提供 `.pen` 文件，使用 `open_document` 传入 `'new'` 创建新的 `.pen` 文件
- **禁止因缺少 `.pen` 文件而跳过设计步骤**

## 4. 实现

- **Pencil 设计稿是代码实现的首要依据**，不是可选参考
- 逐区块对照设计稿还原：布局结构、区块顺序、信息层级、首屏重点、间距节奏、色彩方案
- 转换为 uni-app + Vue 3 + UnoCSS + wot-design-uni 代码
- 实现时校验核心设计规范与信息克制规则
- 允许因组件库限制或平台差异做微调，但必须在验收汇报中逐项标注偏差及原因
- **禁止**在实现阶段自行"优化"设计稿的布局、增加设计稿中没有的区块、或删除设计稿中存在的区块

## 5. 验收

- 页面与业务场景和交互优先级匹配
- 页面没有冗余描述文案
- 最终 UI 仍像企业业务系统
- **设计还原度**：代码实现是否逐区块对照了 Pencil 设计稿，偏差项是否已标注
- 重构模式：最终页面与参考页面在视觉和结构上明显不同

## 6. 实现后调整

代码交付后用户要求调整时，按幅度自动判断路径：

- **小调整**（文案、间距、颜色、字号、单个组件微调）→ 直接修改代码
- **大调整**（区块增删、布局结构变化、信息层级重组、首屏重心改变）→ 先回 Pencil 重新设计，用户确认后再改代码

关键规则：
1. Agent 必须先判断并告知用户属于哪一类
2. 用户可覆盖判断（如明确说"直接改代码"）
3. 多个小调整累积效果等同大调整时，主动建议回 Pencil 统一调整
