# 移动端 UI 生成器 — 核心 Prompt 规则 v5.0

> **分工**：视觉规范（Token / 组件 / 平台约束）看 `design-spec.md`；**设计弹药**（视觉词汇 / 布局原型 / 风格人格）看 `design-arsenal.md`；**C 级满分范例**看 `visual-prompt-examples/`。

## 1. 核心目标：产出 C 级视觉指令型 prompt

调用 Pencil 前，必须把需求整理成 **C 级视觉指令型 prompt**（内容 + 视觉层次 + 焦点 + 间距节奏全锁死，Pencil 几乎只需照着画）。C 级定义见 `design-arsenal.md §0`。

**三管齐下**（缺一不可）：
- **弹药库**（主干）：从 `design-arsenal.md` 取视觉词汇配方、布局原型、风格人格。
- **范例**：对照 `visual-prompt-examples/` 对应页面类型的满分范例仿写。
- **展开追问清单**：对每个区块走 `pencil-prompt-template.md` 的「⭐ C 级 5 问」。

**两条前置纪律**：
- **方向选择前置**：先用 `ui-direction-template.md` 给 ≥3 个差异化方案，用户选定前不写 prompt/代码。
- **Pencil 必经**：选定方案后必须写 C 级 prompt → 调 Pencil 出图 → 用户确认满意，才进代码。选定方案 ≠ 同意设计。

## 2. ⭐ C 级自检清单（生成 prompt 后逐条核对，缺则补）

```
□ 1. 首屏有唯一视觉重心区块，并标注它占屏比例（如 ~60%）
□ 2. 每个区块都答全了「C 级 5 问」：尺寸/内距 · 背景(底色+圆角档+阴影档) · 内部元素字号档+字重档+色彩 · 焦点 · 间距节奏
□ 3. 有明确视觉焦点（金额/关键指标），用大字号+重字重+品牌色/状态色三件套
□ 4. 区块间距疏密交替（不是全页均匀），制造呼吸
□ 5. 所有视觉属性 $token(数值) 双标注，无裸数值硬编码
□ 6. 用真实业务数据（¥125,000 / 张三 / 采购合同-2024089），不用「标题文本」
□ 7. 色彩 80%+ 灰白底 + 局部品牌色点睛，无彩虹图标/处处彩底
□ 8. 标注了页面层级（一级/二级）对应的圆角/装饰/头部规范（page-hierarchy.md）
```

> 任何一条不满足，prompt 还停留在 A/B 级，必须补齐再调 Pencil。

## 3. ⭐ 子 agent 编排剧本（怎么把 prompt 写到 C 级）

用户选定方案、进入 `build_structured_prompt` 阶段后：

**3.1 能力检测**：判断当前运行环境是否提供「派生子 agent / Task」能力。
- **Claude Code**（有 Agent/Task 工具）→ 走 3.2 真编排。
- **Codex / Cursor 或不支持的环境** → 走 3.3 串行。

**3.2 真编排（Claude Code，可选增强）**：
1. 主 agent 把选定方案按区块拆分（如：头部 / 主角区 / 次要区 / 底部操作）。
2. **为每个区块并行派一个子 agent**，每个子 agent 依据 `design-arsenal.md` + 对应 `visual-prompt-examples/` 范例，对该区块走「⭐ C 级 5 问」，产出该区块的 Token 双标注视觉指令。
3. 主 agent 汇总各区块产出，拼成完整结构化 prompt。
4. 主 agent 做**全局节奏校验**：统一跨区块的间距节奏、确保唯一视觉重心、对照 §2 自检清单。
5. 用合并后的 prompt 调 Pencil。

**3.3 串行（其他端 / 不支持子 agent）**：
- 单 agent 对每个区块**依次**走「⭐ C 级 5 问」，自己填满。
- 同样最后对照 §2 自检清单逐条校验。

**3.4 殊途同归**：无论真编排还是串行，最终产物都必须是**通过 §2 C 级自检清单的完整 prompt**。子 agent 只是 Claude Code 上的加速/深挖手段，不是 C 级的前提——不支持就退回串行，质量标准不变。

## 4. Pencil 专属规则与规范引用

- 描述时用语义化容器（`Section` / `Card`），明确信息密度、交互锚点（按钮/搜索/筛选位置）。
- 一级页面标注微信胶囊安全距离。
- **禁止在 prompt 中重复定义颜色/圆角/间距数值的来源**——用 `$token(数值)` 引用 `design-spec.md` 的 Token，不另造。

## 5. 重构模式 Prompt 策略

用户要求"重构"或提供参考图/现有代码时：
- **提取**：仅提取参考页的业务模块、字段、状态、操作。
- **重组**：用 `design-arsenal.md §2` 选**与参考页本质不同**的布局原型，彻底打破原布局。
- **差异化**：≥3 个不同侧重的方向（用 `pencil-prompt-template.md §6` 重构模板）。
- **Pencil 必经**：对原 UI 不满意，更要先看到新设计效果。

## 6. 表单 / 列表 / 详情专项规则（代码实现阶段自动推断）

> 这些是「设计稿 → 代码」阶段的自动推断规则，prompt 阶段也据此组织字段。

- **表单**（`/mug:*:form`）：字段清单（类型/必填/组件/校验/键盘/联动）；校验类型与键盘按 `design-spec.md §5.7` 自动推断；字段 > 6 自动分组（§5.9）；联动按 §5.8。对照范例 `visual-prompt-examples/form.md`。
- **列表**（`/mug:*:list`）：列表项字段映射（主标题/副信息/状态/金额/时间）；展示方式与布局变体按 `§6.7/§6.8` 自动推断；操作模式 A 内联（高频）/ B 左滑（低频）按 §6.4；空/无结果/失败状态处理。对照 `visual-prompt-examples/list.md`。
- **详情**（`/mug:*:detail`）：信息分组 + 三层头部（§7.5）；字段交互（phone 拨号 / 编号复制 / url 跳转）按 §7.4；附件按 §7.6；关联数据 ≤3 条按 §7.7。对照 `visual-prompt-examples/detail.md`。

## 7. 通用交互规则（代码实现阶段）

- **导航**：列表→详情用 `navigateTo` 只传 id；表单提交后 `uni.$emit('list-refresh')`+`navigateBack`；列表 `onLoad` 监听刷新、`onUnload` 清理（`design-spec.md §8.12`）。
- **错误处理**：表单/列表请求 try/catch，失败给具体反馈（toast 或 EmptyState），禁空 catch；按钮 loading 防重复（§8.13）。
- **反馈文案**：成功「{动词}成功」、失败「{动词}失败，{原因}」、确认「确定{动词}{对象}？」，禁笼统文案（§8.2）。
- **骨架屏**：匹配页面类型（表单/列表/详情各自骨架），禁通用骨架（§8.3.1）。

## 8. 反 AI 生成感

正向手法见 `design-arsenal.md §1`（层次/焦点/节奏/点睛）与 `§3` 各人格「反面」；禁止套路清单见 `design-spec.md §9`。核心三条：
- 首屏必须有视觉重心（不做全页等分网格 / 2×2 完美对称）。
- 区块大小有主次、间距有节奏（不均匀排列）。
- 大面积灰白 + 局部品牌色点睛（不处处彩底、不彩虹图标）。
